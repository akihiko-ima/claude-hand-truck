## アーキテクチャパターン

### パイプライン＋レイヤードアーキテクチャ

リアルタイム映像処理の特性上、**映像処理パイプライン**と**レイヤードアーキテクチャ**を組み合わせます。

```
┌────────────────────────────────────────────────────┐
│  入力レイヤー (Input Layer)                          │
│  CameraManager  ←─── カメラデバイス                  │
├────────────────────────────────────────────────────┤
│  検出レイヤー (Detection Layer)                      │
│  HandDetector + CalibrationManager                  │
│  カメラ座標 → テーブル正規化座標への変換               │
├────────────────────────────────────────────────────┤
│  ビジネスロジックレイヤー (Business Logic Layer)      │
│  GridTracker                                        │
│  グリッド管理・累積時間計算・清掃完了判定              │
├────────────────────────────────────────────────────┤
│  出力レイヤー (Output Layer)                         │
│  HeatmapRenderer + DisplayController                │
│  ヒートマップ生成・画面表示                           │
├────────────────────────────────────────────────────┤
│  データレイヤー (Data Layer)                         │
│  DataStorage                                        │
│  セッションデータの永続化                             │
└────────────────────────────────────────────────────┘
```

#### 各レイヤーの依存関係

```
CameraManager
    ↓
HandDetector ← CalibrationManager
    ↓
GridTracker
    ↓        ↘
HeatmapRenderer  DataStorage
    ↓
DisplayController
```

**依存方向の原則**:
- 上位レイヤーから下位レイヤーへの一方向依存のみ許可
- DataStorage はどのレイヤーからも呼び出し可能（横断的関心事）
- MediaPipe/OpenCVへの直接依存は各専門コンポーネント内に閉じる

## データ永続化戦略

### ストレージ方式

| データ種別 | ストレージ | フォーマット | 理由 |
|-----------|----------|-------------|------|
| キャリブレーション設定 | ローカルファイル | JSON | 人間が読める・設定のコピー移行が容易 |
| 清掃セッションデータ | ローカルファイル | JSON | PoCフェーズはシンプルなファイル管理で十分 |
| ヒートマップ画像 | ローカルファイル | PNG | 可逆圧縮・透過対応 |

### ディレクトリ構造

```
{プロジェクトルート}/
├── config/
│   ├── calibration_0.json   # カメラ0のキャリブレーション設定
│   └── calibration_1.json   # カメラ1のキャリブレーション設定
└── data/
    └── sessions/
        └── 20260314_100000/
            ├── session.json   # セッションデータ（清掃率・グリッド状態）
            └── heatmap.png    # 最終ヒートマップ画像
```

### バックアップ戦略

- **頻度**: フレームごとに `session.json` を上書き保存（都度保存）
- **世代管理**: セッションID（日時）でディレクトリ分割のため自動的に世代管理
- **復元方法**: `data/sessions/{session_id}/session.json` を直接読み込む

## パフォーマンス要件

### レスポンスタイム

| 操作 | 目標時間 | 測定環境 |
|------|---------|---------|
| 1フレームの全処理（検出→追跡→描画） | 33ms以内(30fps相当) | CPU Core i5相当、メモリ8GB |
| ヒートマップ更新（GridTracker → 画面反映） | 1秒以内 | 同上 |
| アプリケーション起動（カメラ映像表示まで） | 10秒以内 | 同上 |
| セッションデータ保存（JSON書き込み） | 100ms以内 | SSD環境 |

### リソース使用量

| リソース | 上限 | 理由 |
|---------|------|------|
| メモリ | 2GB | MediaPipeモデルとOpenCVバッファの合計。一般的なPC環境で問題ない範囲 |
| CPU | 80% | 2カメラ処理時の目安。余力を残してOS操作を妨げない |
| ディスク | 1GB/日 | セッションデータとPNG画像の蓄積量（1日30セッション想定） |

### 2カメラ逐次処理の性能試算

**前提条件**: CPU Core i5相当（シングルスレッド処理）

| 処理ステップ | 推定処理時間 |
|------------|------------|
| カメラ1フレーム取得（OpenCV） | 約2ms |
| MediaPipe手検出（1カメラ分） | 約10〜15ms |
| カメラ2フレーム取得 + MediaPipe手検出 | 約12〜17ms |
| GridTracker更新 + HeatmapRenderer | 約1〜2ms |
| DisplayController表示 | 約2〜3ms |
| **合計（2カメラ逐次処理）** | **約27〜39ms** |

**結論**: 処理時間の中央値は約33ms（=30fps相当）であり、パフォーマンス要件を**ほぼ達成可能**と判断する。ただし以下の条件を前提とする:

- MediaPipeはCPUモードで動作（GPUアクセラレーションなし）
- カメラバッファによる追加遅延（1〜2フレーム）を許容する
- 処理が39msを超えた場合（約25fps）は、30fps要件の緩和またはマルチスレッド化を検討する

## セキュリティアーキテクチャ

### データ保護

- **暗号化**: 不要（清掃記録はPII非含有のため）
- **アクセス制御**: `config/` および `data/` ディレクトリはローカルOSのファイルシステムパーミッションに委ねる
- **機密情報管理**: APIキー等の機密情報なし（PoCはローカル完結のため）

### 入力検証

- **バリデーション**: キャリブレーション設定ファイル読み込み時にスキーマ検証を実施
- **サニタイゼーション**: ファイルパスにユーザー入力を含めない（セッションIDは日時から自動生成）
- **エラーハンドリング**: スタックトレースはログファイルに記録し、画面には日本語の簡潔なメッセージのみ表示

---

## スケーラビリティ設計

### データ増加への対応

- **想定データ量**: 1日30セッション × 30日 = 約900セッション（約900MB）
- **パフォーマンス劣化対策**: セッションデータはディレクトリ分割のためI/Oスキャンが発生しない設計
- **アーカイブ戦略**: PoCフェーズは手動削除。将来的に30日以上前のセッションを自動アーカイブ

### 機能拡張性

- **センサーデバイス追加**: `HandDetector` と同一インターフェースを持つ検出クラスを追加することで対応
- **分析項目追加**: `GridTracker` の `CLEAN_THRESHOLD_SECONDS` を設定ファイルから変更可能にすることで対応
- **複数テーブル対応**: `table_id` をセッションデータに保持しているため、テーブルIDによるフィルタリングで対応可能

## テスト戦略

### ユニットテスト

- **フレームワーク**: pytest
- **対象**:
  - `GridTracker`: 累積時間の加算ロジック・清掃完了判定・境界値テスト
  - `CalibrationManager`: ホモグラフィ変換の精度検証
  - `DataStorage`: JSONの読み書き整合性・異常終了シミュレーション
- **カバレッジ目標**: コアビジネスロジック(GridTracker, CalibrationManager)で80%以上

### 統合テスト

- **方法**: 事前収録した動画ファイルを入力として使用（実カメラ不要）
- **対象**:
  - 動画ファイル入力 → GridTracker更新 → DataStorage保存の一連フロー
  - 2カメラの検出結果統合（論理和）の正確性

### E2Eテスト

- **ツール**: 手動テスト（PoCフェーズ）
- **シナリオ**:
  - 実カメラで手を5秒以上かざし、グリッドが清掃済みになること
  - 全24セル完了時にアラートが表示されること
  - qキー終了後にsession.jsonとheatmap.pngが生成されること

## 技術的制約

### 環境要件

- **OS**: Windows 10/11 または Linux (Ubuntu 22.04+)
- **最小メモリ**: 8GB（MediaPipeモデルのロードに4GB以上推奨）
- **必要ディスク容量**: 初期インストール 2GB + データ蓄積分
- **必要な外部依存**: USBカメラ（UVCプロトコル対応）1〜2台

### パフォーマンス制約

- MediaPipeの手検出はGPUアクセラレーションなしの場合CPU依存
- 2カメラ同時処理時は逐次処理（シングルスレッド）で実装し、必要に応じてマルチスレッド化を検討
- OpenCVのカメラキャプチャバッファ遅延により最大1〜2フレームの遅延が発生する可能性あり

### セキュリティ制約

- 清掃映像はローカルPCにのみ保存し、ネットワーク送信は行わない
- PoCフェーズのため認証機能なし（ローカル運用前提）

## 依存関係管理

| ライブラリ | 用途 | バージョン管理方針 |
|-----------|------|-------------------|
| opencv-python | 映像処理・表示 | マイナーバージョンまで許可 (`>=4.0,<5.0`) |
| mediapipe | 手検出 | 最新安定版に追従（モデル精度向上のため） |
| numpy | 数値演算 | メジャーバージョン固定 (`>=2.0,<3.0`) |
| matplotlib | ヒートマップ生成 | マイナーバージョンまで許可 (`>=3.0,<4.0`) |
| pytest | テスト | パッチバージョンのみ自動 |

**方針**:
- `uv.lock` ファイルで全依存関係を固定し再現性を確保
- セキュリティアップデートは適宜 `uv lock --upgrade-package` で個別対応
- MediaPipeはモデルの改善が頻繁なため、最新安定版への追従を推奨
